## Overview

The admin dashboard provides platform administrators with tools to manage users, approve professor applications, and monitor platform statistics. Access is restricted to users with admin privileges.

## Access control

Only authenticated users with the `admin` flag set to `true` can access the admin dashboard.

### Admin authentication check

From `pages/admin.php:6-20`:

```php
if(!IsConnected()){
    header('Location: connection.php');
    exit();
}

$sql = 'SELECT Utilisateurs.id, Utilisateurs.prenom, Utilisateurs.admin 
        FROM sessions 
        INNER JOIN Utilisateurs ON Utilisateurs.id = sessions.user_id 
        WHERE sessions.token = :token';
$request = $db->prepare($sql);
$request->bindParam(':token', $_COOKIE['user_token']);
$request->execute();
$result = $request->fetch(PDO::FETCH_ASSOC);

if(!isset($result['admin']) || $result['admin'] == false){
    header('Location: ../index.php');
    exit();
}
```

<Warning>
  Users without admin privileges are immediately redirected to the homepage when attempting to access admin pages.
</Warning>

### Approve professor request

The approval process upgrades the user role and generates an API token from `pages/admin.php:36-61`:

```php
function AddTeacher($user){
    global $db, $result;
    
    // Update user role to professor
    $sql = "UPDATE Utilisateurs SET Utilisateurs.role = 'professeur' WHERE id = :id;";
    $request = $db->prepare($sql);
    $request->bindParam(':id', $user);
    $request->execute();
    
    DeleteRequest($user);

    // Generate API token
    $token = generateToken();
    $currentDate = date('Y-m-d H:i:s');
    $sql = "INSERT INTO KeyTable (actif, token, date_creation) 
            VALUES (true, :token, :date)";
    $request = $db->prepare($sql);
    $request->bindParam(':token', $token);
    $request->bindParam(':date', $currentDate);
    $request->execute();

    $keyId = $db->lastInsertId();

    // Link API key to user
    $sql = 'UPDATE Utilisateurs SET Utilisateurs.key_id = :kid WHERE Utilisateurs.id = :id';
    $request = $db->prepare($sql);
    $request->bindParam(':kid', $keyId);
    $request->bindParam(':id', $_POST['user']);
    $request->execute();

    header("Location: " . $_SERVER['PHP_SELF']);
}
```

<Steps>
  <Step title="Update role">
    User's role field is changed from 'etudiant' to 'professeur' in the database.
  </Step>
  
  <Step title="Generate API token">
    A unique token is generated for the professor to access API features.
  </Step>
  
  <Step title="Store token">
    The token is saved in the KeyTable with active status.
  </Step>
  
  <Step title="Link to user">
    The key_id is associated with the user's account.
  </Step>
  
  <Step title="Delete request">
    The professor application is removed from DemandeProf table.
  </Step>
</Steps>

## Database schema

### DemandeProf table

```sql
CREATE TABLE DemandeProf (
    id INT PRIMARY KEY AUTO_INCREMENT,
    id_utilisateur INT NOT NULL,
    presentation TEXT NOT NULL,
    date_demande DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (id_utilisateur) REFERENCES Utilisateurs(id) ON DELETE CASCADE
);
```

### KeyTable

```sql
CREATE TABLE KeyTable (
    key_id INT PRIMARY KEY AUTO_INCREMENT,
    actif BOOLEAN DEFAULT TRUE,
    token VARCHAR(255) UNIQUE NOT NULL,
    date_creation DATETIME NOT NULL
);
```

### Utilisateurs admin fields

```sql
ALTER TABLE Utilisateurs ADD COLUMN admin BOOLEAN DEFAULT FALSE;
ALTER TABLE Utilisateurs ADD COLUMN role VARCHAR(20) DEFAULT 'etudiant';
ALTER TABLE Utilisateurs ADD COLUMN key_id INT;
ALTER TABLE Utilisateurs ADD FOREIGN KEY (key_id) REFERENCES KeyTable(key_id);
```

## Security considerations

<Accordion title="Authorization checks">
  - Every admin page starts with `IsConnected()` check
  - Admin flag is verified from database, not trusted from cookies
  - Unauthorized users are redirected before any operations
  - Uses prepared statements to prevent SQL injection
</Accordion>

<Accordion title="CSRF protection">
  While the current implementation uses POST forms, consider adding:
  - CSRF tokens in forms
  - Token validation before processing actions
  - Session-based token generation
</Accordion>

<Accordion title="Input validation">
  - User IDs are validated as integers
  - Action values are checked ("1" for accept, "0" for reject)
  - Database constraints prevent invalid data
</Accordion>

## Admin workflow

<Steps>
  <Step title="Login as admin">
    Admin must authenticate with an account that has admin flag set to true.
  </Step>
  
  <Step title="View dashboard">
    See platform statistics: total users, new users (7 days), new courses (7 days).
  </Step>
  
  <Step title="Review applications">
    Check pending professor applications in the requests table.
  </Step>
  
  <Step title="Evaluate CV">
    Read the presentation text submitted by the applicant.
  </Step>
  
  <Step title="Make decision">
    Click "Accepter" to approve or "Refuser" to reject the application.
  </Step>
  
  <Step title="Confirmation">
    Page refreshes and the processed request is removed from the list.
  </Step>
</Steps>

<Info>
  When a professor is approved, they gain access to the course creation dashboard and receive an API token for advanced features.
</Info>

## Best practices for admins

- Review professor applications carefully before approval
- Monitor new user and course statistics regularly
- Check for suspicious activity patterns
- Maintain a log of approved/rejected applications (future enhancement)
- Only grant admin access to trusted team members

<Warning>
  Admin actions are permanent. Rejecting a professor request deletes the application permanently. Users would need to resubmit if rejected by mistake.
</Warning>
